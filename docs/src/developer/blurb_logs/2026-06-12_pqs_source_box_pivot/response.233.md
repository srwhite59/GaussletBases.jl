## Direct answers

1. Existing support-row/support-descriptor generators without WL/QW coefficient
   matrices:

   - `src/cartesian_shellification/terminal_geometry.jl`
     - `raw_terminal_geometry(...)` emits geometry-only terminal regions with
       `outer_box`, optional `inner_exclusion_box`, `support_count`, and a
       disjoint coverage audit.
     - It emits two `:atom_local_core` regions, optional `:midpoint_slab`
       regions, and `:shared_molecular_shell` regions for bond-aligned H2.
   - `src/cartesian_shellification/CartesianShellification.jl`
     - `shellify(...)` wraps raw geometry into typed `TerminalRegion` records
       carrying `OwnedSupport` and `ShellificationRegion`.
   - `src/cartesian_route_core/shellification_regions.jl`
     - `OwnedSupport`, `owned_cpb`, `owned_cpb_union`,
       `complete_shell_support`, `ShellificationRegion`, and `support_count`
       own route-neutral parent support.
   - `src/cartesian_terminal_lowering/region_contracts.jl`
     - `_direct_terminal_contract(...)` and `_pqs_complete_shell_contract(...)`
       choose direct or PQS lowering contracts for terminal regions, but this
       is per-terminal-region and does not yet group H2 physical support units.
   - `src/pqs_multilayer_shell_region_plan.jl`
     - `pqs_multilayer_shell_region_plan(...)` and
       `_pqs_multilayer_region_support_indices(...)` generate support indices
       from shellification/lowering, but the current plan requires exactly one
       direct core. It does not fit H2 atom-contact core grouping as-is.
   - `src/cartesian_nested_diatomic.jl`
     - `_nested_bond_aligned_diatomic_atom_growth_anatomy(...)` and
       `_nested_bond_aligned_diatomic_atom_growth_construction_plan(...)`
       generate left/right atom boxes, an optional contact cap, and regular
       shared molecular shells with explicit support indices. This is geometry
       and support-only, not coefficient-matrix authority, but it is older
       atom-growth planning vocabulary.

   Not authority for this pass:

   - `bond_aligned_diatomic_nested_fixed_source(...)` and the fake-PQS adapter
     path in `src/pqs_source_box_diatomic_complete_core_shell.jl` expose
     support indices, but they are tied to the old fixed-source sequence and
     `source.sequence.coefficient_matrix`. They should not be reused as
     independent PQS support authority.

2. `atom_contact_core = 275` can be derived from route geometry.

   For the H2 R=4, q=5 target with parent dimensions `(9, 9, 15)` and atom
   centers on the bond axis at indices `5` and `11`, the support is:

   ```text
   left atom core      = 5 * 5 * 5 = 125
   right atom core     = 5 * 5 * 5 = 125
   contact/midpoint slab = 5 * 5 * 1 = 25
   total = 275
   ```

   It is represented or representable as:

   - newer terminal shellification:
     two `:atom_local_core` regions plus one `:midpoint_slab`;
   - older atom-growth anatomy:
     `:left_atom_box`, `:right_atom_box`, and `:contact_cap`.

   The missing route-owned step is the grouping that aliases those primitive
   terminal regions into one `:atom_contact_core` support unit without adding
   retained-transform claims.

3. `shared_shell_1 = 578` and `shared_shell_2 = 362` can be generated by
   shellification-style geometry, but the exact order must be deliberate.

   The atom-contact inner box is `5 x 5 x 11`, so:

   ```text
   inner shared shell = 7 * 7 * 13 - 5 * 5 * 11 = 637 - 275 = 362
   outer shared shell = 9 * 9 * 15 - 7 * 7 * 13 = 1215 - 637 = 578
   ```

   The newer `raw_terminal_geometry(...)` emits shared molecular shells
   inside-out: first `362`, then `578`. The H2 target vocabulary wants
   outside-in order:

   ```text
   :shared_shell_1 => 578
   :shared_shell_2 => 362
   ```

   The older atom-growth construction plan already orders regular shared
   molecular shells outside-in. A new independent route support plan should use
   shellification-owned support but explicitly sort/group shared shells
   outside-in by containment/outer-box volume, not by raw terminal emission
   order.

4. Ownership recommendation:

   - Primitive parent support belongs to `CartesianShellification` /
     `CartesianRouteCore.OwnedSupport`.
   - Lowering contracts belong to `CartesianTerminalLowering`.
   - The three-unit H2 physical support vocabulary should be a compact private
     route-owned object, because `:atom_contact_core` is a union across
     multiple terminal regions and the shared-shell order is target-specific.

   Recommended internal boundary name:

   ```julia
   _PQSIndependentH2PhysicalSupportRegionPlan
   ```

   Recommended fields:

   ```text
   status
   blocker
   support_order
   support_regions
   support_counts
   support_indices
   parent_axis_counts
   nuclear_indices
   bond_axis
   coverage
   authority
   summary
   metadata
   ```

   This object should report `authority =
   :cartesian_shellification_route_geometry` or similar when available. It
   should still report no retained counts, no final dimension, and no physics.
